## Directory Purposes

**`backend/`:**
- Purpose: FastAPI server, market data, database, business logic
- Contains: Python code, tests, dependencies, configuration
- Key files: `pyproject.toml` (dependencies), `app/market/` (core subsystem)

**`backend/app/`:**
- Purpose: Application package; all production code lives here
- Contains: market subsystem, routes (planned), utilities
- Key principle: Everything under `app/` is bundled in wheel; nothing outside this directory is deployed

**`backend/app/market/`:**
- Purpose: Market data abstraction, implementations, and SSE streaming
- Contains: Complete, tested implementation with two data sources (Simulator, Massive)
- Exports: PriceUpdate, PriceCache, MarketDataSource, create_market_data_source, create_stream_router
- Status: Fully implemented and tested

**`backend/app/routes/`:**
- Purpose: Placeholder for FastAPI route handlers (portfolio, watchlist, chat)
- Currently: Empty
- To Do: Implement endpoints listed in PLAN.md section 8
- Pattern: Each major resource (portfolio, watchlist, chat) gets its own router file

**`backend/tests/`:**
- Purpose: Unit tests following pytest conventions
- Organization: Mirrors app structure (tests/market/ mirrors app/market/)
- Run: `uv run --extra dev pytest -v`
- Coverage: Current tests cover market data subsystem only

**`planning/`:**
- Purpose: Shared documentation, contracts, and specifications for all agents
- Key files: `PLAN.md` is the source of truth; all design decisions documented here
- Access: Read-only reference; agents update this only via formal design reviews

**`frontend/` (not yet created):**
- Purpose: Next.js TypeScript project (static export)
- Will contain: React components, Tailwind CSS, SSE client, API calls
- Independence: Complete build process; built separately before being copied into Dockerfile stage 2
- Entry: `frontend/page.tsx` or similar (Next.js conventions)

## Naming Conventions

**Files:**
- Python modules: `snake_case.py` (e.g., `market_data_source.py`)
- Test files: `test_<module>.py` (e.g., `test_cache.py`)
- Static files: lowercase (e.g., `styles.css`, `index.html`)
- Directories: `snake_case` for Python packages, lowercase for assets

**Functions & Classes:**
- Classes: `PascalCase` (e.g., `PriceUpdate`, `SimulatorDataSource`)
- Functions: `snake_case` (e.g., `create_market_data_source()`)
- Private functions: `_snake_case` leading underscore (e.g., `_generate_events()`)
- Constants: `UPPER_CASE` (e.g., `DEFAULT_DT`, `SEED_PRICES`)

**Variables:**
- Loop variables: `i`, `j`, `k` for indices; descriptive otherwise
- Private attributes: `_leading_underscore` (e.g., `self._prices`, `self._lock`)
- Config: `UPPER_CASE` for environment variables and constants

## Where to Add New Code

**New API Endpoint (Portfolio, Watchlist, Chat):**
- Primary code: Create `backend/app/routes/<resource>.py` (e.g., `portfolio.py`)
  - Follow pattern from `backend/app/market/stream.py` (factory function returning APIRouter)
  - Use dependency injection for PriceCache, database connection, MarketDataSource
  - Example: `def create_portfolio_router(db: Database, cache: PriceCache) -> APIRouter: ...`
- Tests: Create `backend/tests/<resource>/test_*.py`
- Registration: In `backend/app/main.py` startup, include the router: `app.include_router(create_portfolio_router(...))`

**New Market Data Source (e.g., real-time WebSocket provider):**
- Implementation: Create `backend/app/market/<provider>_client.py`
- Pattern: Inherit from `MarketDataSource` abstract base in `backend/app/market/interface.py`
- Required methods: `start()`, `stop()`, `add_ticker()`, `remove_ticker()`, `get_tickers()`
- Update factory: Modify `backend/app/market/factory.py` to detect new provider via environment variable
- Tests: Create `backend/tests/market/test_<provider>.py`

**Database Layer (Schema, Queries, Migrations):**
- Schema definitions: Create `backend/db/schema.sql` or implement as migrations in `backend/db/migrations/`
- Query logic: Create `backend/app/db.py` or `backend/app/database/` package
- Pattern: Use async context manager for transactions; explicit query builders or ORM (aiosqlite recommended for SQLite)
- Tests: Create `backend/tests/test_database.py`

**Shared Utility or Helper:**
- Location: `backend/app/utils.py` (if single file) or `backend/app/utils/` package
- Example: Validation functions, formatting, error handling utilities
- Tests: `backend/tests/test_utils.py`

**Frontend Component (Later):**
- Location: `frontend/app/components/<feature>/` (following Next.js structure)
- Example: `frontend/app/components/watchlist/ticker_card.tsx`
- Tests: Co-located `<file>.test.tsx` or in `frontend/__tests__/`
- Styling: Component-scoped Tailwind or CSS modules

## Special Directories

**`backend/.venv/`:**
- Purpose: Virtual environment (generated by `uv sync`)
- Generated: Yes (by uv)
- Committed: No (in .gitignore)
- Do not edit manually; let uv manage

**`backend/.pytest_cache/`:**
- Purpose: Pytest internal cache
- Generated: Yes (by pytest)
- Committed: No (in .gitignore)
- Safe to delete

**`backend/__pycache__/`:**
- Purpose: Compiled Python bytecode
- Generated: Yes (by Python interpreter)
- Committed: No (in .gitignore)
- Safe to delete

**`db/`:**
- Purpose: Runtime SQLite database storage location
- Generated: Yes (SQLite file created by backend on first run)
- Committed: `.gitkeep` only (database file is .gitignored)
- Volume mounted in Docker for persistence across container restarts

**`planning/.codebase/`:**
- Purpose: Auto-generated codebase analysis documents (ARCHITECTURE.md, STRUCTURE.md, etc.)
- Generated: Yes (by gsd-map-codebase agent)
- Committed: Yes (for reference by other agents)
- Pattern: Read-only; overwritten on each analysis run

**`planning/archive/`:**
- Purpose: Archived design documents from completed phases
- Committed: Yes (historical reference)
- Pattern: Docs moved here after phase completion via gsd-cleanup
