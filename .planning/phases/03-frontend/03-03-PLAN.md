<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| backend 400 detail → user-facing toast | error strings from /api/portfolio/trade could leak internals if rendered raw |
| LLM/chat response → DOM render | assistant text + action details rendered in chat bubbles/chips |
| user trade/chat input → backend POST | quantity/ticker/message sent to backend mutations |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-03-08 | Information Disclosure | TradeBar error rendering | high | mitigate | Backend `detail` is mapped to fixed UI-SPEC copy via the substring table; raw `detail` is NEVER rendered (V7). Generic fallback for unmatched errors. |
| T-03-09 | Tampering (XSS) | ChatMessage / ChatPanel rendering | high | mitigate | All assistant text + chip values render via `{value}` JSX (React auto-escapes); NO `dangerouslySetInnerHTML` (V5). Chat prompt-injection trades are re-validated server-side (defense in depth). |
| T-03-10 | Input Validation | TradeBar quantity/ticker | medium | mitigate | Client validates qty>0 + uppercase-alphanumeric ticker (UX); backend `execute_trade` re-validates as the security boundary (V5). |
| T-03-11 | Information Disclosure | Chat disabled probe | low | mitigate | Disabled-state probe uses GET /api/health (no secret exposure); OPENROUTER_API_KEY stays server-side, never sent to client (V6). |
| T-03-SC | Tampering | npm package usage (recharts in charts) | high | mitigate | Packages already audited + pinned in 03-01 (RESEARCH Package Legitimacy Audit all-clear); no new installs in this plan. |
</threat_model>
