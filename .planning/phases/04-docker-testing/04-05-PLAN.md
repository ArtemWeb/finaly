<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| compose network -> app/playwright services | The two test containers communicate over an internal compose network by service name |
| test volume -> production volume | A misconfigured volume could let tests write to real data |
| test image build -> product image | The compose builds the same Dockerfile under test, inheriting its secret-handling |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-04-10 | Tampering | test data isolation | medium | mitigate | app service mounts a dedicated finally-test-data volume, never the production finally-data; LLM_MOCK=true means no real OPENROUTER key is needed or used in E2E (acceptance criteria enforce both) |
| T-04-11 | Information Disclosure | E2E env / secrets | high | mitigate | The compose app service sets only LLM_MOCK=true; no OPENROUTER_API_KEY is passed (mock path needs none), so no secret enters the test environment or image layer |
| T-04-04 | Tampering | playwright/base images | low | accept | mcr.microsoft.com/playwright is Microsoft-published (official); python/node base images via the product Dockerfile are Docker-library official; digest pinning is ASVS-1 nice-to-have; accepted for a demo |
</threat_model>
