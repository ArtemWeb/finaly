<objective>
Build the chat service that turns a user message into a fully-handled chat turn: gather portfolio + watchlist + history context, call the LLM client, auto-execute any returned trades and watchlist changes through the existing validated paths, persist the turn, and return a structured result.

Purpose: This is the orchestration brain of the chat feature. It is the only place that knows how to assemble context and apply LLM-decided actions. Keeping it DB-aware but FastAPI-free makes the whole turn unit-testable without HTTP. Covers CHAT-01 (full portfolio context in prompt), CHAT-03 (trade auto-exec with error reporting), CHAT-04 (history memory + watchlist auto-exec), CHAT-05 (persist + return).
Output: backend/app/chat_service.py, backend/tests/test_chat_service.py.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| LLM output -> trade execution | Model-decided trades/watchlist changes auto-execute against real portfolio state with no confirmation dialog |
| user_message -> SQL persistence | Free-form user text and assistant text written to chat_messages |
| chat_service -> portfolio/watchlist DB | Mutations to cash, positions, watchlist driven indirectly by model output |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-02-04 | Elevation of Privilege | LLM-decided trades auto-execute | high | mitigate | Trades route through the SAME execute_trade validation as manual trades (sufficient cash for buy, sufficient shares for sell, positive quantity, known side). A malicious/hallucinated trade cannot bypass balance checks; TradeError is captured and reported, never silently applied. Auto-exec without dialog is an accepted design decision (simulated money, zero stakes per planning/PLAN.md lines 321-328). |
| T-02-05 | Tampering / Injection | user_message + assistant text -> SQLite | high | mitigate | All chat_messages writes use `?` parameterized aiosqlite queries (never f-string interpolation), matching the Phase 1 pattern in portfolio_service.py/db.py — prompt-injection text cannot break out into SQL |
| T-02-06 | Tampering (prompt injection) | user_message embedded in LLM prompt | medium | accept | Prompt injection can at most cause the model to emit trades/watchlist changes; those are still bounded by execute_trade validation (T-02-04) and the bounded ChatResponse schema. Out-of-scope actions are impossible because only trades + watchlist changes are interpreted. Residual risk accepted for a single-user simulated environment. |
| T-02-07 | Information Disclosure | portfolio context in prompt / chat history | low | accept | Single-user app (hardcoded DEFAULT_USER_ID); no cross-user data. Portfolio context is the user's own data. Accepted. |
</threat_model>
